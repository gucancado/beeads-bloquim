criar modalidades de prazo para tarefas

objetivo:
expandir o conceito atual de "prazo" (data única) para permitir diferentes janelas de execução da tarefa

1) alteração de nomenclatura

- substituir o rótulo "prazo" por "fazer"
- ao lado do rótulo "fazer", adicionar um dropdown para selecionar a modalidade

2) modalidades de prazo

implementar 3 modalidades:

2.1) modalidade "até" (comportamento atual)

- exibir apenas o campo:
  - data final (prazo)

- comportamento:
  - a tarefa pode ser executada a qualquer momento até a data definida
  - após a data, a tarefa permanece executável, porém em atraso
  - não há controle de início por janela de data

- manter o comportamento atual de dependências:
  - se a tarefa tiver tarefas anteriores ligadas a ela
  - e todas essas tarefas anteriores estiverem com status:
    - concluído OU
    - cancelado
  - a tarefa muda automaticamente para "em andamento"

- essa verificação já existe hoje no sistema e deve continuar funcionando normalmente
- a nova funcionalidade de modalidades de prazo não deve quebrar nem alterar essa lógica já existente

2.2) modalidade "entre"

- exibir campos:
  - início (apenas data)
  - prazo (apenas data)

- comportamento:

caso NÃO haja tarefas anteriores ligadas:
- se status = "pendente" e data atual >= início:
  - mudar automaticamente status para "em andamento"

caso HAJA tarefas anteriores ligadas:
- a tarefa só pode iniciar quando:
  - todas as tarefas anteriores estiverem:
    - concluídas OU
    - canceladas
  E
  - data atual estiver dentro da janela:
    - >= início
    - <= prazo

- ao atender essas condições:
  - status muda automaticamente para "pendente" ou "em andamento" conforme a regra já existente de transição entre dependências e liberação de execução
  - respeitar a lógica atual do sistema, que já verifica relações entre tarefas a cada mudança de status

observação:
- não criar uma lógica paralela isolada
- integrar essa modalidade ao mecanismo já existente de atualização automática de status com base nas relações entre tarefas

2.3) modalidade "em"

- exibir apenas o campo:
  - prazo (apenas data)

- comportamento de dados:
  - ao definir/alterar o prazo:
    - campo "início" deve ser automaticamente igual ao prazo

- comportamento funcional:
  - idêntico à modalidade "entre"

- diferença:
  - apenas simplificação visual (um único campo)

3) regra de granularidade de data

- os campos "início" e "prazo" trabalham apenas com data, sem hora
- não exibir seletor de hora no frontend
- não exigir hora no backend
- toda lógica deve considerar somente o dia

interpretação prática:
- "data atual >= início" significa que chegou o dia de início
- "data atual <= prazo" significa que ainda está dentro do dia final
- tratar comparações de forma consistente para evitar erro por timezone ou horário local

4) regras de transição de status

- status devem ser atualizados automaticamente com base em:
  - modalidade selecionada
  - datas configuradas
  - dependências (tarefas anteriores)

- o sistema já verifica relações entre tarefas a cada mudança de status de uma tarefa
- essa verificação existente deve continuar sendo usada como base
- integrar as modalidades novas a esse mecanismo já existente, sem duplicar regras

- garantir que:
  - não haja loops ou múltiplas atualizações desnecessárias
  - mudanças sejam idempotentes

- importante:
  - o sistema não deve bloquear alterações manuais de status por parte do usuário
  - esse comportamento já existe hoje e não deve ser alterado
  - ou seja, mesmo que a tarefa esteja fora da janela de execução, o usuário ainda pode manualmente colocá-la como "em andamento", "concluída", etc.
  - as modalidades de prazo devem influenciar apenas as mudanças automáticas de status, nunca impedir ações manuais do usuário

5) backend

- suportar novos campos:
  - startAt (início)
  - dueAt (prazo)
  - scheduleMode ("ate" | "entre" | "em")

- mesmo que os nomes dos campos sejam startAt e dueAt, o valor funcional usado deve representar apenas a data
- lógica de transição deve ser centralizada no backend (não depender apenas do frontend)

- garantir compatibilidade com tarefas existentes:
  - tarefas antigas devem ser interpretadas como modalidade "até"

6) frontend

- atualizar UI da tarefa:
  - label "fazer"
  - dropdown de modalidade
  - renderização condicional dos campos (início/prazo)

- os inputs de início e prazo devem ser de data, sem hora

- as mudanças visuais devem ser aplicadas de forma consistente em:
  - tela de edição de tarefa
  - lista de tarefas
  - nó do mapa

- garantir consistência de funcionamento e comportamento em todos os contextos:
  - tela de edição de tarefa
  - lista de tarefas
  - nó do mapa (incluindo edições inline)

- qualquer alteração feita em um desses contextos deve refletir corretamente nos demais

- garantir boa usabilidade:
  - validação de datas (início <= prazo)
  - feedback visual de status

7) regras adicionais

- não alterar comportamento de tarefas já concluídas ou canceladas
- garantir consistência em:
  - lista de tarefas
  - nós do mapa
  - tela de edição

8) fora de escopo

- não implementar notificações
- não implementar automações externas

implementar apenas essa lógica
